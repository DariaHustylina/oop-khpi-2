# Лабораторна робота №1
## Тема: Реєстрація заявок на ремонт техніки
## Опис предметної галузі

Система призначена для обліку та обробки заявок на ремонт техніки.
Клієнти створюють заявки з описом проблеми, а майстер виконує ремонт.

### Кожна заявка містить:

унікальний ідентифікатор;

опис проблеми;

статус виконання;

результат ремонту;

клієнта;

техніку, яку ремонтують.

### Основні класи:

RepairItem - одиниця техніки для ремонту

Customer - клієнт сервісу

ServiceOrder - заявка на ремонт

## Опис класів:

### RepairItem:

Описує техніку для ремонту.

#### Поля:

serialNumber : String

isFixed : boolean

#### Методи:

RepairItem(String serialNumber)

RepairItem(String serialNumber, boolean isFixed)

markAsFixed()

toString(), equals(), hashCode()

### Customer:
Описує клієнта сервісу.

#### Поля:

customerId : String

fullName : String

#### Методи:

Customer(String customerId, String fullName)

toString()


### ServiceOrder:
Описує заявку на ремонт.

#### Поля:

orderId : String

description : String

status : String

result : String

customer : Customer

item : RepairItem

#### Методи:

ServiceOrder(String orderId, String description, Customer customer, RepairItem item)

startRepair()

finishRepair()

toString()

## Демонстрація роботи
### У класі Main реалізовано:

створення об’єктів Customer та RepairItem

створення заявки ServiceOrder

показ життєвого циклу заявки: нова -> в процесі -> завершена

демонстрація методів toString(), equals(), hashCode()

## Висновок:
У ході виконання лабораторної роботи було розроблено спрощену модель системи реєстрації заявок на ремонт техніки. Було виділено основні сутності предметної галузі: RepairItem, Customer та ServiceOrder, які відображають техніку, клієнта та заявку відповідно.

Для перевірки рівності об’єктів у класі RepairItem використовується поле serialNumber, оскільки воно є унікальним ідентифікатором кожного пристрою. Саме тому два об’єкти вважаються рівними, якщо їх серійні номери співпадають, незалежно від інших полів.

Розроблені класи можуть взаємодіяти між собою наступним чином: один клієнт може створювати декілька заявок, а кожна заявка пов’язана з конкретною одиницею техніки. У подальшому модель можна розширити, додавши можливість зберігання історії ремонтів, призначення майстрів, облік часу виконання робіт та інтеграцію з базою даних.

Таким чином, створена модель є базовою, але придатною для подальшого розвитку в наступних лабораторних роботах.
